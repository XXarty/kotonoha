# KOTONOHA 视觉改版与内容扩充设计

**日期：** 2026-07-15  
**状态：** 用户已确认视觉、页面结构、内容方案与技术验收标准  
**目标：** 将现有 MVP 改造成素雅、清晰、温情的日语学习网站，并在可追溯、可验证的前提下把公开内容扩展到约 10,000 个词汇和 120 个语法单元。

## 1. 背景与已确认决策

当前网站已上线并具备首页、词汇、语法、五十音、搜索、复习、账户与 Neon 进度同步。公开内容由静态 JSON 随 GitHub 和 Vercel 部署，Neon 只保存用户数据。当前 2,000 个词汇和 30 个语法单元是人为设置的 MVP 上限，不代表公开来源的全部覆盖。

本轮已确认以下决策：

- 视觉主方向采用「晨雾来信」，叠加「和纸书房」的细小点状纹理。
- 文字必须清晰，温情但不过度抒情，不使用动漫、樱花、鸟居等刻板日式装饰。
- 动效轻、慢、克制，并完整遵守 `prefers-reduced-motion`。
- 搜索不再作为独立导航页面；改为从全站顶部展开的当前页搜索层。
- 内容扩充采用“分层精选”路线，目标约 10,000 个词汇和 120 个语法单元。
- 质量优先于凑数；严格验证后不足目标时发布真实数量。
- 公开内容继续放在 GitHub 静态快照中，Neon 继续只保存账户、收藏和学习进度。

## 2. 范围

### 2.1 本轮包含

- 重做全站视觉 Token、背景、字体、间距、卡片、按钮和状态样式。
- 重写首页、目录、详情、空状态、登录提示和来源说明的主要文案。
- 重做 Header、移动端导航和全站展开式搜索。
- 重组首页、词汇目录、语法路径和详情页的信息层级。
- 生成并懒加载轻量搜索索引。
- 将词汇管线扩展到最多约 10,000 个经过验证的词条。
- 将语法数据模型和课程扩展到 120 个完整单元。
- 增加内容质量统计、拒绝统计、确定性构建和回退保护。
- 完成单元、组件、端到端、响应式、无障碍和生产构建验证。

### 2.2 本轮不包含

- 不导入来源或许可不明确的 JLPT 分级表。
- 不为了达到精确数量而保留歧义中文释义或自动生成虚假释义。
- 不为全部 10,000 个词条自动编造例句。
- 不接入 OCR、PDF 派生内容、音频、手写识别、社交或付费功能。
- 不把公共词汇和语法正文迁移到 Neon。
- 不在本轮接入 Tatoeba；其双语例句作为后续独立增强评估。

## 3. 视觉设计系统

### 3.1 气质

网站像每天收到一封简短、安静的学习来信。视觉重点是舒展留白、温和纸张感、清楚的阅读层级和少量自然色，不做传统文化主题展板，也不做冷硬的数据后台。

### 3.2 颜色

建议 Token：

| Token | 建议值 | 用途 |
| --- | --- | --- |
| `paper` | `#F7F3EA` | 暖米白主背景 |
| `mist` | `#EDF1E8` | 雾绿渐变与分区背景 |
| `ink` | `#26332E` | 主文字 |
| `ink-soft` | `#65716B` | 次要说明 |
| `moss` | `#667D6E` | 主按钮、链接与选中状态 |
| `sage` | `#BCC8BC` | 轻边框、标签和装饰 |
| `line` | `#D7DDD5` | 分隔线 |
| `white-soft` | `rgba(255,255,255,.62)` | 半透明内容面板 |

所有最终颜色必须以实际渲染结果通过 WCAG AA 对比度检查；建议值可在实施阶段微调。

### 3.3 字体

- 日文标题、词条主字和展示标题使用日文衬线字体栈。
- 中文正文、导航、按钮和表单使用高可读无衬线字体栈。
- 数据标签可使用克制的等宽字体，但不以全大写英文主导页面。
- 正文默认不小于 16px，移动端行高保持约 1.7。

### 3.4 背景与材质

- 页面使用雾绿到暖米白的极浅渐变。
- 和纸点纹由纯 CSS 小点构成，低透明度、固定尺寸、不参与读屏。
- 纹理不得覆盖输入框、正文卡片和需要高对比度的区域。

### 3.5 动效

- 搜索层：约 220ms 淡入并向上移动 6–10px。
- 卡片：悬停或聚焦时最多上移 2px，不使用大幅缩放。
- 今日一词：首页首屏以轻微透明度和位移进入，避免逐字炫技。
- 路径与列表：只做颜色、边框和小位移反馈。
- `prefers-reduced-motion: reduce` 时关闭非必要动画、平滑滚动和位移。

## 4. 文案设计

### 4.1 原则

- 用陪伴式语气替代命令式语气。
- 保持信息明确，不用诗意文字掩盖功能。
- 首屏负责建立温度，目录和详情负责提高学习效率。
- 数量、来源、许可和错误信息必须准确直接。

### 4.2 示例方向

- 首页主标题：“今天，也为自己留一点语言的时间。”
- 首页说明：“从一个词、一句话开始，让日语慢慢住进日常。”
- 主要行动：“开始今天的学习”或“收下今日一词”。
- 空状态：“这里暂时还没有内容。换一个关键词，或回到学习路径看看。”
- 复习提示：“不用一次记住全部。把今天会的，轻轻留下来。”

最终文案应集中管理或至少使用一致的语气清单，避免不同页面各自形成风格。

## 5. 页面与信息架构

### 5.1 全站 Header

- 左侧为 `ことのは` 品牌入口。
- 主导航保留单词、语法、五十音和复习。
- 搜索改为带文字或图标的紧凑按钮，随时可打开搜索层。
- 账户入口保持可见，但视觉权重低于学习入口。
- 移动端使用可访问的折叠菜单；搜索按钮仍可直接触达。

### 5.2 全站搜索层

- 从当前页面顶部展开，不切换路由。
- 支持日文、假名、罗马字和中文释义。
- 首次打开时懒加载紧凑搜索索引；本次会话后续使用缓存结果。
- 默认最多显示 8 条结果，每条显示类型、日文、读音、中文短释义和目标路由。
- 支持 `Escape` 关闭、上下方向键选择、`Enter` 打开和明确焦点管理。
- 加载失败时显示重试按钮以及词汇、语法目录入口。
- 旧 `/search` 地址保留兼容重定向，但不再作为导航页面。

### 5.3 首页

首页顺序：

1. 温情主标题、简短说明和“开始今天的学习”。
2. 今日一词，使用真实静态内容。
3. 继续学习或登录后恢复进度。
4. 单词、语法、五十音三条入口。
5. 真实内容数量和来源提示。

首屏不得预加载完整词库。

### 5.4 词汇目录

- 顶层分为“日常核心”和“进阶扩展”。
- 支持词性、来源优先标记、假名索引和分页。
- 每页约 60–80 条，避免渲染数千个节点。
- 列表优先显示日文、假名和中文短释义；英文和罗马字降为辅助信息。
- 所有数量由当前启用快照计算，不写死目标值。

### 5.5 语法路径

分成四条学习路径：

1. 基础：句子结构、形容词、动词和基本助词。
2. 核心：时态、连接、条件、愿望、请求和授受。
3. 常用表达：推测、比较、原因、目的、变化和口语表达。
4. 进阶：被动、使役、敬语、正式表达和复杂句型。

目录显示顺序、完成状态、单元数量和预计阅读时间。预计时间采用明确、统一的简单规则生成，不伪装为真实用户统计。

### 5.6 内容详情

词汇详情包含日文、假名、罗马字、词性、中英文释义、来源、内容版本和可选例句。没有可靠例句时整个例句区不渲染。

语法详情包含表达、接续、中文解释、至少一个日中双语例句、常见误区、相近表达或区别说明、来源链接、内容版本和许可证。

## 6. 词汇扩充设计

### 6.1 数量与分层

- 目标是最多约 10,000 条经过验证的发布记录。
- “日常核心”优先使用 JMdict 的常用或优先标记。
- “进阶扩展”可包含其他可靠候选，但不展示无法证明的频率排名或 JLPT 等级。
- 构建后不足 10,000 条时发布实际数量，不降低验证标准。

### 6.2 匹配规则

当前主要按日文写法查找 Kaikki 释义。本轮升级为：

1. 规范化 JMdict 写法、读音和词性。
2. 规范化 Kaikki 词条写法、可用读音和词性信息。
3. 优先按“写法 + 读音”唯一匹配。
4. 缺少读音时，只允许能够通过写法和词性消除歧义的匹配。
5. 多义集合无法稳定对应时进入拒绝统计。
6. 中文释义为空、异常或包含无法清理的章节标记时拒绝。

### 6.3 记录字段

词汇记录保留稳定 `id`，并增加或明确：

- `tier`: `core` 或 `extended`
- `priority_tags`: 来自 JMdict 的可验证标记
- `examples`: 可选的可靠例句数组
- `content_version`
- `source_id`、`source_key` 和中文释义来源

现有稳定 ID 不改变，Neon 收藏和进度继续可解析。

## 7. 语法扩充设计

### 7.1 课程规模

- 总目标为 120 个单元。
- 四条路径各自拥有连续顺序，但全局仍保留稳定 `display_order`。
- 结构参考 Tae Kim 的基础、核心、特殊表达和高级主题。
- 中文说明重新组织和改写，不复制许可不明确的第三方中文翻译。

### 7.2 单元字段

语法模型从单个例句升级为结构化字段：

- `expression`
- `connection`
- `explanation_zh`
- `examples[]`，至少一个日中双语例句
- `common_mistakes[]`，至少一条明确提醒
- `related_entries[]`，保存稳定语法 ID
- `source_url`
- `license_key`
- `content_version`
- `path` 与 `display_order`

原创例句标记为 `kotonoha-original`。所有条目必须通过字段完整性、稳定 ID、顺序、关联存在性和来源 URL 验证。

## 8. 构建与数据流

离线内容流程：

1. 下载固定版本的允许来源并记录 URL、日期和 SHA-256。
2. 将 JMdict 与 Kaikki 转成内部候选模型。
3. 执行规范化、唯一匹配、分层、拒绝统计和字段验证。
4. 验证 120 个语法单元及其关联。
5. 生成词汇、语法、假名、来源、验证结果和轻量搜索索引。
6. 在临时目录验证文件数量、哈希和引用。
7. 全部通过后原子替换已提交快照。
8. GitHub 提交经过验证的静态输出，Vercel 构建不得下载上游数据。

搜索数据流：

`构建器 -> search-index.json -> 用户首次打开搜索层时加载 -> 客户端规范化匹配 -> 最多 8 条结果 -> 进入现有详情路由`

## 9. 组件边界

- `design tokens`：颜色、字体、间距、圆角、阴影和动效。
- `SiteHeader`：导航、账户入口和搜索触发器。
- `GlobalSearch`：索引懒加载、输入、键盘交互、结果和错误状态。
- `HomeHero`：温情文案和主要行动。
- `DailyWord`：稳定每日内容展示。
- `VocabularyDirectory`：分层、筛选、索引和分页。
- `GrammarPath`：四条路径和学习顺序。
- `EntryDetail`：正文、学习操作、来源和可选内容块。
- `content repository`：启用来源过滤、详情解析、分页和稳定路由。
- `content builders`：上游解析、匹配、验证和原子输出。

组件只依赖明确的输入模型，不直接读取上游格式。视觉组件不负责内容质量判断，内容构建器不负责页面表现。

## 10. 错误处理与降级

- 搜索索引加载失败：显示重试、词汇目录和语法目录入口。
- 搜索无结果：显示温和但明确的空状态，不生成伪结果。
- Neon 不可用：公开首页、目录、详情和搜索继续工作；账户操作显示具体错误。
- 上游下载、哈希或解析失败：构建退出，保留线上快照。
- 词汇不足目标：发布真实验证数量，并记录拒绝统计。
- 来源禁用：目录、搜索、今日一词和复习补全均过滤该来源。
- 旧进度无法解析：不展示已移除正文，但不删除用户进度。

## 11. 无障碍与性能

- 正文与交互颜色达到 WCAG AA。
- 所有操作支持键盘并有清晰焦点。
- 搜索层打开后焦点进入输入框，关闭后回到触发按钮。
- 360px 宽度无页面级横向滚动，触控目标不小于 44px。
- 首页不加载完整词库；搜索索引仅在首次打开时加载。
- 大列表分页或分段渲染，避免数千个 DOM 节点。
- 搜索输入采用短防抖并限制结果数量。
- 所有非必要动画在减少动态设置下关闭。

## 12. 测试与验收

### 12.1 内容测试

- JMdict 与 Kaikki 的写法、读音和词性匹配。
- 缺失中文、同形异义、重复候选和异常字段进入拒绝统计。
- 核心与扩展分层只使用可追溯来源字段。
- 120 个语法单元字段完整、顺序稳定、关联有效。
- 同一输入重复构建生成字节一致的 JSON。
- Manifest 数量、文件哈希、搜索索引和实际记录一致。
- 来源关闭后所有公开读取路径都过滤该来源。

### 12.2 应用测试

- 内容仓库的目录、分页、详情、来源过滤和稳定 ID。
- 搜索索引规范化与日文、假名、罗马字、中文匹配。
- 搜索层打开、关闭、焦点、键盘选择、无结果和加载失败。
- 首页、词汇路径、语法路径、详情、收藏和复习组件。
- 旧 `/search` 地址兼容跳转。

### 12.3 端到端与视觉验收

- 首页到今日一词学习。
- 当前页打开搜索、选择结果并进入详情。
- 词汇分层、筛选和分页。
- 语法路径到详情和学习评分。
- 登录用户收藏和复习同步。
- 360、768、1024、1440 像素宽度检查。
- 键盘导航、减少动态和基础读屏语义检查。
- Python 测试、Vitest、lint、类型检查、生产构建和关键 Playwright 冒烟测试全部通过。

## 13. 实施顺序

1. 使用 Sites 设计能力校准页面结构和设计选择。
2. 使用 frontend-design 落实视觉 Token、排版、动效和温情文案。
3. 使用 Build Web Apps 在现有 Next.js 架构中实现组件、搜索和响应式行为。
4. 扩展词汇数据模型、匹配器、验证器和搜索索引。
5. 扩展并验证 120 个语法单元。
6. 集成内容、运行完整测试并做视觉检查。
7. 提交 GitHub，先做 Vercel Preview，确认后再更新 Production。

## 14. 完成标准

- 已确认的视觉方向、顶部搜索层和温情文案在全站一致呈现。
- `/search` 不再是主导航页面，旧地址仍兼容。
- 词汇发布数量尽可能接近 10,000，且所有条目通过验证。
- 语法恰好 120 个完整单元并通过关联验证。
- 公开浏览不依赖 Neon。
- 许可证、来源、哈希和拒绝统计可追溯。
- 所有约定测试和生产构建通过。
- Vercel Preview 经人工确认后才更新正式站。
